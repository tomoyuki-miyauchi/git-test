# GitとGitHubの違い

初心者がよく混同しますが、

GitとGitHubは別のものです。

| Git            | GitHub         |
| -------------- | -------------- |
| バージョン管理ツール     | Gitリポジトリ共有サービス |
| ローカルPCで動作する    | インターネット上で動作する  |
| コミットやブランチを管理する | リモートリポジトリを管理する |

今回学んできた内容はすべて Git の機能です。

GitHubはそのGitリポジトリを共有するためのサービスです。

---

# Gitは分散型バージョン管理システム

Gitは **Distributed Version Control System (DVCS)** と呼ばれます。

つまり、

> 開発者全員がリポジトリの完全なコピーを持つ

という仕組みです。

例えば3人チームの場合、

```text
GitHub
   │
   ├── 開発者A
   ├── 開発者B
   └── 開発者C
```

ではなく、

```text
GitHub (共有用)

開発者A ← 完全な履歴
開発者B ← 完全な履歴
開発者C ← 完全な履歴
```

という構造になります。

そのためインターネットが無くてもコミットやブランチ操作が可能です。

---

# 共同開発の基本的な流れ

実際の開発では次のような流れになります。

```text
1. リポジトリを取得する
2. ブランチを作成する
3. 機能を開発する
4. GitHubへアップロードする
5. レビューを受ける
6. mainへマージする
```

---

# Clone

最初にリポジトリを自分のPCへコピーします。

```bash
git clone <repository-url>
```

この操作を **Clone** と呼びます。

今回の勉強会でも最初に実施しました。

---

# Push

ローカルで作成したコミットをGitHubへ送信します。

```bash
git push origin main
```

これを **Push** と呼びます。

Pushすると他のメンバーも変更内容を確認できるようになります。

---

# Pull

他の開発者が追加した変更を取得します。

```bash
git pull
```

これを **Pull** と呼びます。

共同開発では頻繁に実行します。

---

# Pull Request

実際のチーム開発では、

直接 `main` へマージすることはほとんどありません。

代わりに

> 「この変更を取り込んでください」

という依頼を送ります。

これを **Pull Request (PR)** と呼びます。

---

## Pull Requestの流れ

```text
main
  │
  └── feature/high-score
          │
          └── 開発完了
                  │
                  ▼
           Pull Request
                  │
                  ▼
              レビュー
                  │
                  ▼
                Merge
```
